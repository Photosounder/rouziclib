== Threads

`general/threading.h` provides portable thread handles, detached threads, joins, priorities, yielding, mutexes, semaphores, and 32-bit atomics. Thread functions use an `int (*)(void *)` signature.

[source,c]
----
typedef struct
{
    volatile int32_t quit;
    rl_thread_t handle;
} worker_t;

static int worker_main(void *ptr)
{
    worker_t *worker = ptr;

    // Keep doing work until the owner requests shutdown
    while (!rl_atomic_load_i32(&worker->quit))
        do_one_job();

    return 0;
}

static int worker_start(worker_t *worker)
{
    // Reset the stop flag before publishing the worker
    rl_atomic_store_i32(&worker->quit, 0);
    return rl_thread_create(&worker->handle, worker_main, worker);
}

static void worker_stop(worker_t *worker)
{
    // Signal shutdown before waiting for the thread
    rl_atomic_store_i32(&worker->quit, 1);

    // Join before freeing data visible to the worker
    rl_thread_join_and_null(&worker->handle);
}
----

`rl_thread_create()` returns `1` on success and `0` on failure. The example is intentionally schematic: `do_one_job()` represents application code, and shared job data still needs a mutex, atomics, or a semaphore as appropriate. Spacewar uses joinable threads for analysis tools and detached threads for command processing.

== Mutexes and atomics

Initialize a mutex before publishing it to a worker, lock only around the shared state that needs protection, and destroy it after all users have stopped. Use `rl_atomic_load_i32()`, `rl_atomic_store_i32()`, `rl_atomic_add_i32()`, and `rl_atomic_get_and_set()` for small synchronization flags and counters.

The `RL_EXCL_THREADING` configuration removes the implementation and substitutes inert types, so code that requires real parallel work must not enable that switch.

== Audio bus

The audio layer mixes callback buses into an interleaved stereo floating-point stream. A callback receives the output stream, an `audiosys_t` pointer, its bus index, and the application data pointer registered for that bus. Read `sys-\>buffer_len`, `sys-\>sec_per_sample`, and `sys-\>bus[bus_index].stime` to generate the current block.

[source,c]
----
typedef struct
{
    double frequency;
    double gain;
} tone_t;

static void tone_audio(float *stream, void *sys_ptr, int bus_index, void *data_ptr)
{
    audiosys_t *sys = sys_ptr;
    tone_t *tone = data_ptr;
    int i;
    double t;

    // Handle bus expiry or explicit unregistration
    if (stream == NULL)
        return;

    // Add one mono tone to both output channels
    t = sys->bus[bus_index].stime;
    for (i = 0; i < sys->buffer_len; i++, t += sys->sec_per_sample)
    {
        float sample = (float) (sin(2.0 * pi * tone->frequency * t) * tone->gain);
        stream[i * 2] += sample;
        stream[i * 2 + 1] += sample;
    }
}

static int tone_keep_alive(tone_t *tone)
{
    // Refresh the registration more often than its expiry duration
    return audiosys_bus_register(tone_audio, tone, 1, 0.0);
}
----

Call `sdl_audiosys_init(1024)` once during SDL startup, then call `tone_keep_alive()` every frame while the sound should remain active. An expiry value of `0.0` selects the implementation's two-second default; a bus expires if it is not refreshed within that interval.

When `use_mutex` is nonzero, the audio callback is locked automatically. Lock `audiosys.bus[bus_index].mutex` around main-thread writes to the same application data. Call `audiosys_bus_unregister(tone)` before freeing `tone`; unregistration sends the callback a blocking deinitialization signal with `stream == NULL` and `bus_index == -1`.

The current `sdl_audiosys_init()` device-opening implementation is complete for SDL2. Its SDL3 branch is still marked TODO, so SDL3 applications need their own device setup for now.

== Background work

Keep expensive file, codec, or numerical work off the UI thread, then transfer results to the main thread under a clear ownership and synchronization rule. Spacewar's video sorter and position calculator demonstrate this pattern. Timing and deliberate-pacing functions are covered by link:core-data.adoc[Core data and memory].
